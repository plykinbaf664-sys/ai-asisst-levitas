# AGENTS.md

## Контекст проекта

Проект — Telegram AI-ассистент для управленца.

Ассистент должен помогать руководителю разруливать рабочие процессы через Telegram:

* принимать команды в личном чате;
* структурировать хаотичные поручения;
* отправлять задачи и сообщения в нужные командные чаты;
* фиксировать договорённости;
* хранить историю задач;
* помогать контролировать сроки;
* постепенно расширяться под документы, встречи, аналитику и другие рабочие сценарии.

Главная идея проекта:
не просто чат-бот, который отвечает текстом, а операционный ассистент, встроенный в рабочие Telegram-чаты команды.

---

## Принципы работы агента

Работай как senior full-stack engineer.

Перед любыми изменениями:

1. изучи текущую структуру проекта;
2. пойми уже существующую архитектуру;
3. дай короткий план;
4. перечисли файлы, которые собираешься менять;
5. дождись подтверждения, если задача может затронуть несколько частей проекта.

---

## Главные правила разработки

* Не ломай существующую логику.
* Не делай лишний рефакторинг без запроса.
* Не добавляй новые библиотеки без необходимости.
* Не меняй архитектуру проекта без явного согласования.
* Не трогай env-файлы и секреты.
* Не хардкодь токены, chat_id, URL и ключи.
* Все настройки должны идти через env или конфиг.
* Меняй минимальное количество файлов.
* Одна задача = один понятный результат.
* После изменений всегда объясняй, как проверить работу руками.

---

## Архитектурная логика

Проект может включать несколько слоёв:

* Telegram Bot Layer — приём и отправка сообщений;
* Command Layer — обработка команд пользователя;
* AI Parsing Layer — разбор свободного текста;
* Task Layer — задачи, дедлайны, статусы;
* Storage Layer — хранение чатов, задач, истории;
* Reminder Layer — напоминания;
* Config Layer — настройки сценариев и поведения ассистента.

Не смешивай эти слои без необходимости.

---

## Формат ответа агента

Всегда отвечай так:

1. Что понял по задаче.
2. Короткий план.
3. Какие файлы будут изменены.
4. Что именно будет сделано.
5. Как проверить результат.
6. Риски или ограничения, если они есть.

---

## Что нельзя делать без отдельного запроса

* подключать реальные рабочие чаты;
* менять production webhook;
* деплоить проект;
* удалять данные;
* менять базу данных без объяснения;
* добавлять внешние сервисы;
* менять package.json;
* делать крупный рефакторинг;
* добавлять новые сценарии “на будущее”.

---

## Критерии хорошего результата

Результат хороший, если:

* задача решена минимальным изменением;
* код понятный;
* нет лишних зависимостей;
* нет хардкода секретов;
* можно проверить вручную;
* следующий шаг проекта стал проще;
* текущая архитектура не усложнилась без причины.

---

## Важное правило

AGENTS.md содержит только общие правила проекта.

Конкретные задачи, MVP-ограничения, сроки и текущие приоритеты не нужно зашивать сюда навсегда.
Их нужно получать из отдельного пользовательского промпта в текущей сессии.
